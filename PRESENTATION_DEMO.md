# RedChain Malaysia — Presentation Demonstration

## Recommended preparation

Use three browser profiles or three different browsers:

- **Admin account**
- **Donor account**
- **Recipient account**

Use fictional presentation information only. Do not upload a real identity card or real patient medical information. A simple dummy PDF labelled “Presentation Verification Document” is enough for the workflow demonstration.

- Donor display name: `Aiman H.`

- Recipient display name: `Sara M.`

## Exact demonstration order

### 1. Home page

Show:

- Malaysia-focused landing page
- Live emergency request panel
- Verified donor, request and fulfilment statistics
- Privacy, matching and admin feature cards

Say:

> RedChain is a privacy-aware blood donor management and emergency matching platform for Malaysia. It replaces public phone-number lists and unstructured social-media searching with verified profiles, structured requests and controlled contact sharing.

### 2. Donor registration and profile

Using the donor account:

1. Register as donor.
2. Confirm email and sign in.
3. Open **My Profile**.
4. Set general location, blood group and availability.
5. Show that legal name and phone are in the private section.

Say:

> The public and private profile data are separated. Other members can see only the display name and general location. Full contact information is protected.

### 3. Verification upload

1. Open **Verification**.
2. Upload the dummy PDF/JPG/PNG.
3. Show status `Pending`.

Say:

> Verification evidence is uploaded to a private Supabase Storage bucket. It is available only to the owner and an authorised administrator.

### 4. Administrator review

Using the admin account:

1. Open **Admin Control Centre**.
2. Open the pending document using the signed private link.
3. Approve the donor.
4. Show the verified donor count and user table.

Say:

> The administrator checks the submission and controls verification status. This improves trust and prevents ordinary members from giving themselves a verified badge.

### 5. Emergency request

Using the recipient account:

1. Open **Create emergency request**.
2. Enter patient initials, blood group, component, hospital, location, required time and urgency.
3. Enter requester phone in the private contact section.
4. Submit.

Say:

> The public request contains only operational information. The requester’s phone and email are stored in a separate protected table.

### 6. Donor matching and response

Using the verified donor account:

1. Open **Emergency Requests**.
2. Filter by `O+` and Kuala Lumpur.
3. Open the request.
4. Select **Offer to help** and enter a short message.

Say:

> Verified and available donors can respond to compatible active requests. Blood group and Malaysia location filters reduce manual searching.

### 7. Recipient acceptance

Using the recipient account:

1. Open the request.
2. Show the donor response.
3. Accept the donor.
4. Show the request becoming `Matched`.
5. Show the accepted donor contact section.

Say:

> Contact information is not public. The system reveals the accepted donor’s protected contact details only to the request owner after the controlled workflow allows it.

### 8. Standalone contact consent

1. From the recipient account, open **Find Donors**.
2. Request contact access for another verified donor.
3. From the donor account, open **Contact Requests**.
4. Approve or decline.
5. Return to the recipient account and show the result.

Say:

> RedChain also supports direct donor discovery, but the donor decides whether their contact information can be released.

### 9. Status and notifications

Show:

- Open, matched, fulfilled and cancelled statuses
- Notifications generated by requests, responses and verification decisions
- Dashboard profile completion

Say:

> Structured status tracking prevents fulfilled or cancelled emergency posts from continuing to circulate as if they were active.

### 10. Research and administration

Using the admin account, show:

- Feedback rating averages
- Audit logs
- Request moderation
- Account suspension control
- Announcements

Say:

> The feedback module evaluates usefulness, usability, trust and privacy, matching the research objectives. Audit logs and moderation provide administrative oversight.

## Closing statement

> RedChain demonstrates how software engineering can address a real Malaysian healthcare coordination problem through authentication, role-based access, database design, Row Level Security, private document storage, structured matching and a consent-based privacy layer. It supports coordination, while final donor eligibility and blood compatibility remain the responsibility of authorised hospitals or blood centres.

## Questions the lecturer may ask

### Why use Supabase?

It provides authentication, PostgreSQL, Storage and Row Level Security in one platform, which suits a database-driven web application and allows the prototype to demonstrate real access control.

### Why use Vercel?

It provides direct GitHub deployment for the Next.js front end and automatically rebuilds the website after repository updates.

### How is privacy protected?

Public profile data, private contact data and verification documents are separated. RLS policies and a private Storage bucket restrict access. Phone numbers are not returned by public request or donor-directory queries.

### Can any donor mark themselves verified?

No. Database triggers and RLS policies restrict approval fields to administrators. Donors can submit evidence and move to pending status, but cannot approve themselves.

### Does matching prove medical compatibility?

No. The prototype performs exact blood-group and location filtering for coordination. An authorised hospital or blood centre must confirm donor eligibility and clinical compatibility.

### What is original about the system?

It combines verified donor profiles, Malaysia location filtering, structured emergency requests, protected contact storage, donor-controlled contact release, request status tracking and administrator oversight in one workflow.
